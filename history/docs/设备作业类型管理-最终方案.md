# 设备作业类型管理 - 最终方案总结

## ✅ 已完成的工作

### 1. 简化方案设计

**核心改变**：
- ❌ 删除复杂的"按日期范围批量配置"方案
- ✅ 改为简单的"设备级别作业类型管理"

---

### 2. 数据库调整（services/db.js）

#### 表结构修改
```sql
-- old_supplier_devices 表增加字段
work_type_name VARCHAR(100) DEFAULT '其他' COMMENT '当前作业类型'
```

#### 删除的表
- ❌ `work_type_config` 表（不再需要）

#### 新增函数
```javascript
// 更新设备的作业类型
updateDeviceWorkType(macid, workTypeName)

// 获取设备的当前作业类型  
getDeviceWorkType(macid)

// 批量更新设备作业类型
batchUpdateDeviceWorkType(macids, workTypeName)
```

---

### 3. API接口（index.js）

#### 新增3个API

| 接口 | 方法 | 权限 | 说明 |
|------|------|------|------|
| `/api/device/work-type` | POST | 无需登录 | 农户修改设备作业类型 |
| `/api/devices/list` | GET | 无需登录 | 获取设备列表（含作业类型） |
| `/api/device/work-type/batch` | POST | 超级管理员 | 批量更新设备作业类型 |

#### 删除的API
- ❌ `/api/work-type-configs` (POST/GET/DELETE)
- ❌ `/api/work-type-configs/preview`
- ❌ `/api/work-type-configs/:id/apply`

---

### 4. 数据同步逻辑（services/oldSupplierService.js）

#### 修改前（复杂）
```javascript
// 查询配置表，按优先级获取作业类型
const workTypeInfo = await db.getWorkTypeForRecord(
  'old_api', device.macid, date, device.cooperative_name

```

- ✅ 代码更简洁

- ✅ 逻辑更清晰

## 🎯 工作流程

### 农户修改作业类型

```
1. 农户登录 farmer.html
   ↓
2. 切换到"设备管理"Tab
   ↓
3. 查看设备列表（显示当前作业类型）
   ↓
4. 点击某台设备 → 选择新作业类型
   ↓
5. 调用 POST /api/device/work-type
   ↓
6. 更新 old_supplier_devices.work_type_name
   ↓
7. 下次同步时，自动使用新的作业类型
```

---

### 定时同步数据

```
1. 凌晨4点定时任务触发
   ↓
2. 调用旧供应商API获取面积数据
   ↓
3. 遍历每个设备
   ↓
4. 读取设备的 work_type_name
   ↓
5. 写入 work_records 表（使用该作业类型）
```

## 🔧 使用示例

### 1. 农户修改作业类型

```javascript
// farmer.html 中的代码
async function updateDeviceWorkType(macid, workTypeName) {
  const res = await fetch('/api/device/work-type', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ macid, workTypeName })
  });
  
  if (res.code === 0) {
    alert('✅ 作业类型已更新');
    loadDevices(); // 重新加载设备列表
  }
}
```

---

### 2. 获取设备列表

```javascript
async function loadDevices() {
  const res = await fetch(`/api/devices/list?orgId=${currentUser.org_id}`);
  const devices = res.data || [];
  
  // 渲染设备列表
  devices.forEach(device => {
    console.log(`${device.macid}: ${device.work_type_name}`);
  });
}
```

---

### 3. 管理员批量修改

  -d '{

## ⚠️ 重要说明

### 1. 历史数据不会自动更新

**问题**：修改设备的作业类型后，已有的历史数据不会自动更新。

**解决方案**：
- 方式1：使用数据管理页面（`data-repair-work.html`）手动修改
- 方式2：编写SQL脚本批量更新

```sql
-- 示例：更新某设备的历史数据
UPDATE work_records 
SET work_type_name = '玉米播种'
WHERE t_number = '386934' 
  AND source = 'old_api';
```

---

### 2. 默认值

- 新导入的设备，`work_type_name` 默认为"其他"
- 建议农户登录后先设置作业类型

---

### 3. 数据来源统一

- 北斗设备和旧供应商API是同一个数据源
- 都使用 `source='old_api'`
- 不需要区分两者

## 💡 方案优势

| 特性 | 说明 |
|------|------|
| **简单** | 只需一个字段，无需配置表 |
| **直观** | 农户直接在设备列表中修改 |
| **灵活** | 随时可以修改，立即生效 |

## 🔄 与旧方案对比

| 维度 | 旧方案（按日期配置） | 新方案（设备级别） |
|------|-------------------|------------------|
| 配置粒度 | 全局/合作社/终端 + 日期 | 仅终端级别 |
| 灵活性 | 高（可按时间段） | 中（整体修改） |
| 复杂度 | 高（配置表+优先级） | 低（单字段） |
| 易用性 | 复杂（多个配置项） | 简单（直接修改） |
| 代码量 | ~600行 | ~100行 |
| 适用场景 | 季节性频繁变化 | 作业类型稳定 |

**结论**：新方案更适合实际使用场景，简单易用。
