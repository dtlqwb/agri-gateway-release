# 作业类型批量配置功能说明

## 📋 功能概述

为了解决北斗设备/旧供应商API数据**缺少作业类型**的问题，实现了按日期范围批量配置作业类型的功能。

### 核心特点

✅ **灵活配置**：支持终端、合作社、全局三个级别  
✅ **按日期范围**：可以为不同时间段设置不同的作业类型  
✅ **自动应用**：新同步的数据自动应用配置  
✅ **历史数据**：可以一键应用到已有历史数据  
✅ **优先级机制**：终端 > 合作社 > 全局  

---

## 🎯 使用场景

### 场景1：季节性作业类型变化

**问题**：同一台农机在不同季节做不同的作业
- 3-5月：玉米播种
- 9-11月：玉米收割

**解决方案**：
```
配置1: 2026-03-01 ~ 2026-05-31 | 所有设备 | 玉米播种
配置2: 2026-09-01 ~ 2026-11-30 | 所有设备 | 玉米收割
```

---

### 场景2：不同合作社的作业类型不同

**问题**：不同合作社种植作物不同
- 合作社A：玉米
- 合作社B：小麦

**解决方案**：
```
配置1: 2026-03-01 ~ 2026-12-31 | 合作社A | 玉米播种
配置2: 2026-03-01 ~ 2026-12-31 | 合作社B | 小麦播种
```

---

### 场景3：特定设备的特殊配置

**问题**：某台设备换了新农机，作业类型改变

**解决方案**：
```
配置1: 2026-04-01 ~ 2026-12-31 | 设备386934 | 水稻插秧
```

---

## 🚀 使用方法

### 1. 访问管理页面

登录系统后，点击顶部导航栏的 **"🎯 作业类型"** 标签。

或者直接访问：`http://localhost:3001/work-type-config.html`

---

### 2. 创建新配置

填写表单：

| 字段 | 说明 | 是否必填 |
|------|------|---------|
| 数据源 | 选择数据来源（旧供应商API / 北斗设备） | ✅ |
| 开始日期 | 配置的起始日期 | ✅ |
| 结束日期 | 配置的结束日期 | ✅ |
| 终端号 | 指定设备（留空则应用到所有设备） | ❌ |
| 合作社 | 指定合作社（留空则应用到所有合作社） | ❌ |
| 作业类型 | 选择作业类型名称 | ✅ |

**优先级规则**：
- 如果填写了"终端号" → 优先级=2（最高）
- 如果只填写了"合作社" → 优先级=1
- 如果都没填 → 优先级=0（全局默认）

---

### 3. 预览影响数据

点击 **"👁️ 预览影响数据"** 按钮，可以看到：
- 影响的记录数量
- 前20条记录的详细信息（日期、设备、合作社、面积）

---

### 4. 创建配置

确认无误后，点击 **"✅ 创建配置"** 保存。

---

### 5. 应用到历史数据

在配置列表中，找到刚创建的配置，点击 **"应用"** 按钮。

系统会：
1. 查找符合条件的历史记录
2. 批量更新作业类型字段
3. 显示更新的记录数

⚠️ **注意**：此操作不可撤销，请谨慎操作！

---

## 📊 配置示例

### 示例1：春耕播种配置

```
数据源: 旧供应商API
开始日期: 2026-03-01
结束日期: 2026-05-31
终端号: (留空)
合作社: (留空)
作业类型: 玉米播种
```

**效果**：
- 2026年3-5月的所有旧供应商数据，作业类型自动设为"玉米播种"
- 新同步的数据也会自动应用此配置

---

### 示例2：秋收配置（按合作社）

```
数据源: 旧供应商API
开始日期: 2026-09-01
结束日期: 2026-11-30
终端号: (留空)
合作社: 某某合作社
作业类型: 玉米收割
```

**效果**：
- 只有"某某合作社"的设备在9-11月的数据会被设置为"玉米收割"
- 其他合作社不受影响

---

### 示例3：特定设备配置

```
数据源: 旧供应商API
开始日期: 2026-04-01
结束日期: 2026-12-31
终端号: 386934
合作社: (留空)
作业类型: 水稻插秧
```

**效果**：
- 设备386934从4月开始的所有数据都设为"水稻插秧"
- 即使有其他全局配置，此配置优先级更高

## 🔧 技术实现

### 数据库表结构

```sql
CREATE TABLE work_type_config (
  id INT AUTO_INCREMENT PRIMARY KEY,
  source VARCHAR(50) NOT NULL COMMENT '数据源',
  start_date DATE NOT NULL COMMENT '开始日期',
  end_date DATE NOT NULL COMMENT '结束日期',
  t_number VARCHAR(50) DEFAULT '' COMMENT '终端号',
  org_name VARCHAR(100) DEFAULT '' COMMENT '合作社',
  work_type VARCHAR(50) DEFAULT '' COMMENT '作业类型代码',
  work_type_name VARCHAR(100) NOT NULL COMMENT '作业类型名称',
  priority INT DEFAULT 0 COMMENT '优先级',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 优先级查询逻辑

当同步新数据时，系统按以下顺序查找配置：

1. **终端级别**：`t_number = '386934' AND 日期在范围内`
2. **合作社级别**：`org_name = '某某合作社' AND t_number='' AND 日期在范围内`
3. **全局级别**：`t_number='' AND org_name='' AND 日期在范围内`
4. **默认值**：如果没有匹配的配置，使用"其他"

## 📝 API接口

### 1. 创建配置
```
POST /api/work-type-configs
Authorization: Bearer <token>

{
  "source": "old_api",
  "startDate": "2026-03-01",
  "endDate": "2026-05-31",
  "tNumber": "",
  "orgName": "",
  "workTypeName": "玉米播种",
  "priority": 0
}
```

### 2. 获取配置列表
```
GET /api/work-type-configs?source=old_api
Authorization: Bearer <token>
```

### 3. 删除配置
```
DELETE /api/work-type-configs/:id
Authorization: Bearer <token>
```

### 4. 预览影响数据
```
GET /api/work-type-configs/preview?source=old_api&startDate=2026-03-01&endDate=2026-05-31
Authorization: Bearer <token>
```

### 5. 应用配置到历史数据
```
POST /api/work-type-configs/:id/apply
Authorization: Bearer <token>
```

## ⚠️ 注意事项

1. **配置不会自动应用到历史数据**
   - 新同步的数据会自动应用配置
   - 历史数据需要手动点击"应用"按钮

2. **优先级机制**
   - 终端配置 > 合作社配置 > 全局配置
   - 同一优先级下，最新创建的生效

3. **日期范围重叠**
   - 如果多个配置日期范围重叠，优先级高的生效
   - 建议避免同一级别的重叠配置

4. **删除配置不影响已应用的数据**
   - 删除配置只会影响未来的数据同步
   - 已更新的历史数据不会回滚

## 💡 最佳实践

### 1. 先配置后同步

建议在同步数据前先创建好配置，这样新数据会自动应用正确的作业类型。

### 2. 定期审查配置

每个季度检查一次配置，确保符合作业实际情况。

### 3. 使用预览功能

在应用配置到历史数据前，务必先预览影响的数据，确认无误后再执行。

### 4. 命名规范

建议使用清晰的作业类型名称，如：
- ✅ "玉米播种"、"小麦收割"
- ❌ "作业1"、"类型A"

## 📞 常见问题

### Q1: 为什么新同步的数据还是"其他"？

**A**: 检查是否有匹配的配置：
- 日期是否在配置范围内
- 数据源是否正确
- 是否有更高优先级的配置覆盖

### Q2: 如何修改已应用的配置？

**A**: 
1. 删除旧配置
2. 创建新配置
3. 重新应用到历史数据

或者直接在数据管理页面手动修改。

### Q3: 配置会影响云途安的数据吗？

**A**: 不会。配置是按数据源隔离的，`source='old_api'` 的配置只影响旧供应商数据。
