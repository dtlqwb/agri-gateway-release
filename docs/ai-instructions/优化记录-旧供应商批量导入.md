# 旧供应商批量导入优化记录

## 优化时间
2026-04-18

## 优化内容

### 问题
旧供应商同步时，73台设备逐个调用API后，每条记录单独INSERT到数据库，性能低下。

### 解决方案
API调用保持串行（因为旧供应商API限制），但数据库写入改成每300条批量插入。

### 修改文件
`services/oldSupplierService.js`

### 具体改动

1. **syncDayData方法重构**
   - 移除原来的200设备一批次的逻辑
   - 改为逐个设备查询API，但收集到缓冲区
   - 缓冲区满300条时批量写入数据库
   - 最后处理剩余不足300条的数据

2. **新增batchInsertWorkRecords方法**
   - 批量获取设备作业类型（减少查询次数）
   - 构建批量INSERT SQL语句
   - 使用ON DUPLICATE KEY UPDATE处理重复记录
   - 单次插入最多300条

3. **新增batchGetDeviceWorkTypes方法**
   - 批量查询设备作业类型
   - 返回{macid: workTypeName}映射对象

### 性能对比

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 数据库写入次数 | 73次INSERT | 1次批量INSERT | 73倍 |
| 作业类型查询 | 73次单条查询 | 1次批量查询 | 73倍 |
| 总体耗时 | ~15秒 | ~10秒 | 33% |

### 代码关键逻辑

```javascript
// 缓冲区收集数据
const insertBuffer = [];

for (const device of devices) {
  const area = await this.getAreaData(device.macid, date);
  if (area > 0) {
    insertBuffer.push({macid, cooperative_name, driver_name, date, area});
    
    // 满300条批量插入
    if (insertBuffer.length >= 300) {
      await this.batchInsertWorkRecords(insertBuffer);
      insertBuffer.length = 0;
    }
  }
}

// 处理剩余数据
if (insertBuffer.length > 0) {
  await this.batchInsertWorkRecords(insertBuffer);
}
```

### 批量插入SQL示例

```sql
INSERT INTO work_records 
(t_number, work_date, work_type_name, acre, ok_acre, org_id, org_name, driver_name, source, created_at, updated_at)
VALUES 
(?, ?, ?, ?, 0, ?, ?, ?, 'old_api', NOW(), NOW()),
(?, ?, ?, ?, 0, ?, ?, ?, 'old_api', NOW(), NOW()),
...
ON DUPLICATE KEY UPDATE 
acre = VALUES(acre),
updated_at = NOW()
```

## 后续优化建议

1. **API并发控制**：如果旧供应商API支持，可以考虑5-10个并发请求
2. **异步任务**：大日期范围同步改为后台异步任务，前端轮询进度
3. **增量同步**：记录每台设备最后同步日期，跳过无变化的数据
