### 1. repairMachineData() - 设备数据修复

```sql
WHERE m.source IN ('old', 'old_api')
```

**影响**:

## 💡 根本原因分析

### 为什么会有两个source值？

1. **历史原因**:
   - 早期Excel导入使用 `source = 'old'`
   - 后来API同步使用 `source = 'old_api'`
   - 代码未统一

2. **设计缺陷**:
   - 没有统一的常量定义
   - 硬编码字符串容易出错
   - 缺少数据源管理规范

### 建议的长期解决方案

#### 方案1: 统一数据源标识（推荐）

将所有旧供应商数据统一为 `source = 'old'`:

```javascript
// 在oldSupplierService.js中
await db.runSql(
  `INSERT INTO work_records (... , source) VALUES (..., 'old')`,
  [...]
);

// 然后运行一次性迁移脚本
UPDATE work_records SET source = 'old' WHERE source = 'old_api';
UPDATE machines SET source = 'old' WHERE source = 'old_api';
```

#### 方案2: 定义常量

```javascript
// config/constants.js
module.exports = {
  DATA_SOURCE: {
    YUNTINAN: 'yuntinan',
    OLD_SUPPLIER: 'old',  // 统一使用这个
    OLD_API: 'old',       // 别名，指向同一个值
  }
};

// 使用时
const { DATA_SOURCE } = require('../config/constants');
WHERE source = '${DATA_SOURCE.OLD_SUPPLIER}'
```

#### 方案3: 添加数据源映射层

```javascript
// utils/dataSource.js
function normalizeSource(source) {
  const map = {
    'old': 'old',
    'old_api': 'old',
    'yuntinan': 'yuntinan'
  };
  return map[source] || source;
}

// 查询时自动转换
const normalizedSource = normalizeSource(filter.source);
```

## ⚠️ 注意事项

### 1. 向后兼容性

当前修复使用了 `IN ('old', 'old_api')`，兼容两种数据源。

如果将来统一为 `'old'`，需要：
1. 执行数据迁移SQL
2. 修改所有 `IN ('old', 'old_api')` 为 `= 'old'`
3. 测试验证

### 2. 性能影响

使用 `IN` 比 `=` 稍慢，但对于小数据集（几千条）影响可忽略。

如果需要优化，可以：
- 添加索引: `CREATE INDEX idx_source ON work_records(source)`
- 统一数据源后改回 `=`

### 3. 新数据写入

确保新的API同步也使用正确的source值：

```javascript
// oldSupplierService.js
// 确认这里使用的是 'old_api' 还是 'old'
source: 'old_api'  // 或 'old'，需要统一
```
